# Stratégies de Routage : Routes Router & Modular Router

Nexy propose deux stratégies principales de routage : le **Routes Router** (Basé sur les fichiers) et le **Modular Router** (Basé sur les modules).

## Routes Router (Basé sur les fichiers)

Le **Routes Router** utilise la structure physique du répertoire `src/routes/` pour définir automatiquement les URLs de votre application.

- **Topologie-Driven** : Les dossiers et fichiers correspondent directement aux URLs.
- **Support pour .nexy, .mdx, .py** : Chaque fichier du répertoire `src/routes/` est une route.
- **Routage Dynamique** : Utilisez des crochets `[]` pour les segments dynamiques (ex: `src/routes/[id].nexy` -> `/{id}`).
- **SEO-Friendly** : Chaque route est rendue sur le serveur (SSR).

| Fichier | URL |
| --- | --- |
| `src/routes/index.nexy` | `/` |
| `src/routes/docs/intro.mdx` | `/docs/intro` |
| `src/routes/api/health.py` | `/api/health` |

## Modular Router (Basé sur les modules)

Le **Modular Router** (situé dans `src/app/`) est conçu pour les APIs complexes et les applications d'entreprise. Il utilise des décorateurs de classe pour des déclarations de route explicites.

- **Modularité** : Organisez votre application en `Modules`, `Contrôleurs` et `Fournisseurs`.
- **Injection de Dépendances** : Utilisez le système DI de Nexy pour gérer vos services et ressources.
- **Déclarations Explicites** : Définissez vos routes avec des décorateurs comme `@Controller`, `@Get`, `@Post`, etc.
- **Testabilité** : Testez facilement vos contrôleurs et services de manière isolée.

```python
@Controller("/api/v1/users")
class UserController:
    def __init__(self, user_service: UserService):
        self.service = user_service

    @Get("/{id}")
    async def get_profile(self, id: int):
        return await self.service.get_user(id)
```

## Routage Hybride

La puissance de Nexy réside dans sa capacité à combiner ces deux stratégies :
1.  Utilisez le **Modular Router** pour vos microservices, APIs et logique métier.
2.  Utilisez le **Routes Router** pour vos interfaces utilisateur, contenu et blogs.

Cette séparation permet une conception architecturale propre tout en maintenant la rapidité du développement basé sur les fichiers.

---

**Conseil** : Utilisez le **Routes Router** pour les pages publiques et le **Modular Router** pour les services internes et les APIs.
