# 思维模型：Nexy 的内部运作方式

Nexy 是一个统一的抽象层，它同步了服务器（Python/FastAPI）和界面（HTML/JS）。为了高效地使用 Nexy，理解其编译过程和渲染循环至关重要。

## 1. 编译阶段（Build Time）

执行 `nx dev` 或 `nx build` 时，Nexy 会编译您的源文件（`.nexy`、`.mdx`），并将其转换为可执行的 Python 代码：
- **AST 分析 (Abstract Syntax Tree)**：Nexy 分析文件的逻辑块（Python）以提取元数据（props、组件导入）。
- **导入转换**：组件导入（例如 React、Vue 或其他 `.nexy` 文件）被转换为对内部函数 `Import()` 的调用。该函数处理路径解析和数据序列化。
- **模板生成**：HTML/Jinja2 模板被编译，将 PascalCase 组件标签转换为 Python 函数调用。

## 2. 请求生命周期（Runtime）

以下是访问页面时的技术步骤：
1.  **服务器执行**：Nexy 执行与路由对应的文件的已编译 Python 代码。
2.  **Props 验证**：传递给组件的数据会根据逻辑块中定义的类型通过 **Pydantic** 进行验证。
3.  **SSR 渲染 (Jinja2)**：引擎在服务器上生成最终的 HTML。
4.  **资源注入**：如果配置中启用了 Vite 编排器，`Vite()` 函数（通常放在 layout 中）会注入前端运行所需的脚本。

## 3. SSR 单体渲染（Vite-less）

如果您不使用 JavaScript 框架，Nexy 旨在独立运行，不依赖于 Node.js 或 Vite：
- 界面通过 `.nexy` 服务端组件（Python + HTML）构建。
- 渲染完全由 Python 处理，提供最佳性能和 SEO。
- Vite 生态系统是一个交互扩展，您只能通过 `nexyconfig.py` 启用它。

## 4. 统一路由策略

Nexy 结合了两种互补的代码组织方式：
- **基于文件的路由 (File-Based Routing)**：使用 `src/routes/` 目录结构自动定义入口点（页面、API、WebSocket）。
- **基于模块的路由 (Module-Based Routing)**：为复杂的业务服务和依赖注入使用基于类的架构（模块、控制器）。

---

**Nexy 的思维模型：一个强大的 Python 渲染引擎，它将多语言组件转换为纯 HTML，并通过 Vite 无缝扩展到 JavaScript 生态系统。**
