# Routing Strategies: Routes Router & Modular Router

Nexy offers two primary routing strategies: **Routes Router** (File-Based) and **Modular Router** (Module-Based).

## Routes Router (File-Based)

The **Routes Router** uses the physical file structure of your `src/routes/` directory to automatically define your application's URLs.

- **Topology-Driven**: Folders and files map directly to URLs.
- **Support for .nexy, .mdx, .py**: Each file in the `src/routes/` directory is a route.
- **Dynamic Routing**: Use square brackets `[]` for dynamic segments (e.g., `src/routes/[id].nexy` -> `/{id}`).
- **SEO-Friendly**: Every route is rendered on the server (SSR).

| File | URL |
| --- | --- |
| `src/routes/index.nexy` | `/` |
| `src/routes/docs/intro.mdx` | `/docs/intro` |
| `src/routes/api/health.py` | `/api/health` |

## Modular Router (Module-Based)

The **Modular Router** (located in `src/app/`) is designed for complex APIs and enterprise applications. It uses class-based decorators for explicit route declarations.

- **Modularity**: Organize your application into `Modules`, `Controllers`, and `Providers`.
- **Dependency Injection**: Leverage Nexy's DI system to manage services and resources.
- **Explicit Declarations**: Define routes using decorators like `@Controller`, `@Get`, `@Post`, etc.
- **Testability**: Easily test individual controllers and services in isolation.

```python
@Controller("/api/v1/users")
class UserController:
    def __init__(self, user_service: UserService):
        self.service = user_service

    @Get("/{id}")
    async def get_profile(self, id: int):
        return await self.service.get_user(id)
```

## Hybrid Routing

Nexy's strength lies in its ability to combine these two strategies:
1.  Use the **Modular Router** for your microservices, APIs, and business logic.
2.  Use the **Routes Router** for your user interfaces, content, and blogs.

This separation allows for a clean architectural design while maintaining the speed of file-based development.

---

**Tip**: Use the **Routes Router** for public-facing pages and the **Modular Router** for internal services and APIs.
