# Server vs Client Components

Nexy uses a hybrid component model. To build efficient applications, it's crucial to understand the difference between **Server Components** and **Client Components**.

## Server Components (.nexy, .mdx)

By default, every file in Nexy is a **Server Component**.

- **Execution**: They run exclusively on the server (Python).
- **No Bundle**: They don't send JavaScript to the browser unless they import client components.
- **SSR**: They produce pure HTML that is served directly to the browser.
- **Capabilities**: Direct access to databases, internal APIs, and server-side secrets.

## Client Components (.tsx, .vue, .svelte,)

Client Components are used to add interactivity to your application.

- **Execution**: They are bundled by **Vite** and hydrated on the client (browser).
- **Interactivity**: Use them for state management, event listeners, or browser-only APIs (e.g., `localStorage`).
- **Isolation**: They are isolated islands of interactivity within your server-rendered pages.

## Automatic Orchestration

Nexy automatically detects when a Server Component (`.nexy`) imports a Client Component (`.tsx`, etc.).

1.  **Static Analysis**: During compilation, Nexy identifies client-side imports.
2.  **Vite Activation**: If a client component is detected, Nexy triggers the Vite orchestrator to bundle the assets.
3.  **Automatic Hydration**: Nexy handles the serialization of props and the mounting of the client component in the browser. No manual directives like `client:load` are required.

---

**Tip**: Use Server Components for 90% of your UI and Client Components only for rich interactivity. This keeps your application fast and SEO-friendly.
